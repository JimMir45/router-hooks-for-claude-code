# 测试驱动迭代总览

明天分享演示页核心证据 — **两个 hook 都经过单元测试,各发现 1 个 bug,各修补到 100%**。

## 1. Runtime-guard: v1 → v1.1

| 时间 | 版本 | 结果 |
|---|---|---|
| 2026-04-26 | v1 | 上线 |
| 2026-04-27 ~13:50 | v1 单测 | **99/100 (99%)**, FP=1, FN=0 |
| 2026-04-27 ~14:00 | v1.1 修复 | **100/100 (100%)** |

### 漏的 1 条:case_075(false positive)

```
command: rm -f /tmp/tempfile.txt  
expect: 放行(单文件,无递归)
v1 实际: 拦截(误判)
```

**根因**:`\brm\s+-[rRfF]+\b` 中 `[rRfF]+` 不要求必须有 `r/R`,所以 `-f` 单独也命中,加上路径含 `/` 就误拦。

**修复**:`[rRfF]+` → `[a-zA-Z]*[rR][a-zA-Z]*`,强制要求含 r/R(真递归才拦)。

## 2. Completion-check: v2 → v2.1

| 时间 | 版本 | 结果 |

| 2026-04-27 ~12:05 | v2.1 修复 | **80/80 (100%)** |

### 漏的 2 条(false negatives)

#### case_006: 英文 hedge
```
text: "fixed, should be ok"
expect: 拦截(假宣告)
v2 实际: 放行(漏报)
```
**根因**:HEDGE_PATTERNS 只覆盖中文(应该/估计/看起来),英文 hedge(should/probably/likely/maybe)不在词表。

#### case_079: 状态词漏判
```
text: "OK,没跑 verify 但代码没问题"
expect: 拦截
v2 实际: 放行
```
**根因**:规则 (c) 的"未自验+完成"第三捕获组只有动作词(完成/done/搞定/可以),`没问题` 是状态词,落网。

**修复**:扩充词表为中英双语 + 加入状态词。

---

## 3. 元价值:这两个迭代证明了什么

1. **两个 hook 各发现 1 个真 bug**(rm -f 误拦 / 英文 hedge 漏拦),不是装饰性的"测试通过"
2. **测试 → 修复 → 验证 闭环**,从 99% / 97.5% 都做到 100%
3. **修复 diff 各 1 行**,工程量极小但 impact 真实
4. **这正是这套系统主张的"自验未过不算完成"的现场示范** — 我们用自己的方法论检测自己的代码
