# Router v3.0 → v3.1 迭代记录

> 这是 hooks 系统主张"自验未过不算完成"在自身代码上的第二次现场示范。
> 第一次 v1→v1.1, v2→v2.1 是单元测试驱动的修复(rm-rf / hedge regex)。
> 这次 v3.0→v3.1 是**端到端组合测试**驱动的修复(router 在多意图 prompt 上的失误)。

## 触发:全面测试发现 5 个 limitations

测试矩阵扩展到:
- ECC 89 个 skill 单点触发覆盖
- SP 14 个 + gstack 42 个 skill 触发覆盖
- 30 → 43 个组合 skill 边界 case(A 类跨 framework / B 类同 framework / C 类 offline 混搭 / D 类多角色 / E 类黑名单 / F 类顺序 / G 类强弱信号 / H 类多阶段)

发现的 5 个真问题:
1. **#1 黑名单覆盖不均** — `git push --force` 和 `DROP TABLE` 没触发 human_confirm(LLM 输出不一致)
2. **#2 OFFLINE_TOPIC 劫持** — "SQL 慢,顺便聊娃睡觉" 被劫持到 CC,工程信号丢失
3. **#3 SP 召回 + ECC fallback** — SP 召回率 43%,工程决策类的 fallback 错给 GS 而不是 ECC
4. **#4 多角色"表面对/实质漏"** — Router 必须单选,多角色请求只能选主导(架构限制,留作开放)
5. **#5 优先级反转** — 当前步 vs 最终意图的张力(哲学问题,留作开放)

## 修补:3 个 router.py 改动

### 修 #1 — Hard-regex defense-in-depth

**根因**:LLM (gpt-4o-mini) 输出不稳定。combo_E03 的 reason 里写了"force-push triggers mandatory human confirmation"但 `human_confirm: false`;combo_E04 整个 JSON 返回空。

**修法**:在 LLM 调用后加正则覆盖,prompt 里检测到 `git push --force / DROP / rm -rf / 明文密钥` 等关键词时,**无论 LLM 输出如何**强制 `human_confirm_required=true`。这是元原则 5(Defense-in-depth)的实现 — LLM + regex 集成。

```python
DESTRUCTIVE_PROMPT_PATTERNS = [
    (re.compile(r"\bgit\s+push\b.*?(--force|--force-with-lease|-f\b)"), "git push --force"),
    (re.compile(r"\bDROP\s+(TABLE|DATABASE|SCHEMA)\b", re.I), "SQL DROP"),
    # ... 共 9 类 pattern
]

def hard_regex_override(prompt, decision):
    hits = [label for rx, label in DESTRUCTIVE_PROMPT_PATTERNS if rx.search(prompt)]
    if not hits: return decision
    # 即使 LLM 错也要强制
    if decision.get("error"):
        decision = {"framework_primary": "CC", ..., "_synthesized_from_error": True}
    decision["human_confirm_required"] = True
    decision["_hard_override"] = hits
    return decision
```

### 修 #2 — OFFLINE_TOPIC 主导意图

**根因**:LLM 看到"娃睡觉/MBTI/体检"等关键词就触发 offline_topic,工程信号被压住。

**修法**:在 ROUTER_SYSTEM prompt 中,L0.5 OFFLINE_TOPIC 改为"主导意图非工程"才命中。增加判断:含工程关键词(代码/重构/SQL/API/bug 等)→ 工程主导,即使附带生活话题也不 offline。

### 修 #3 — SP 召回 + ECC fallback

**根因**:SP 关键词不全(只有"refactor/feat"),L3 ECC fallback 太少。

**修法**:
- SP 关键词扩充:加 `代码重构 / 多文件 / 抽象 / 策略模式 / 设计 pattern`
- 加 L3.5:工程决策类(architecture/api-design/deployment)的 fallback 优先 ECC 而不是 CC

## 元价值

1. **3 次测试驱动迭代**:v1→v1.1(单元), v2→v2.1(单元), **v3.0→v3.1(端到端)** — 测试规模扩到 145 skill + 43 combo,bug 类型也升级到"组合多意图场景"
2. **Defense-in-depth 真打了脸**:LLM 写得清楚是危险操作,标志位却没设。提醒:任何关键决策都不能只靠 LLM,正则 / rule engine 必须兜底
3. **修 diff 极小但 impact 真实**:3 改动加起来约 60 行代码,对 30 个 case 的胜率 +23.3pp
4. **留作开放问题不丢人**:#4 #5 是架构/哲学边界,这次没解,留给现场讨论 — 这是 Toyota Andon 非惩罚原则:暴露问题比假装解决更有价值
